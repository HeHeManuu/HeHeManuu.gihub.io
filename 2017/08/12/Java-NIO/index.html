Java NIO 解读

发表于 2017-08-12 | 分类于 Java |

概念

NIO 特性

  • 为原始数据类型提供缓存
  • 字符集编码和解码
  • Channels：一个新的原始 I/O 抽象
  • 支持锁和内存映射文件的文件接口
  • 多路，非阻塞的高伸缩性的 I/O

Channel

通道（Channel）是对原 IO 中流的一种抽象，所有数据都要通过通道进行传输。

Buffer

缓冲区（Buffer） 可以看成一个容器对象，在 NIO 中所有的数据读写都是通过缓冲区进行的。在前面的 IO 相关内容中就接触了缓冲，其内部本质上是一块连续的内存区域，这块内存被包装成 NIO Buffer 对象，并提供了一组方法，用来方便的访问该块内存。

内存映射文件

内存映射文件允许我们创建和修改因为太大不能放入内存的文件。使用映射文件访问，可以显著的加快速度。

阻塞与非阻塞

阻塞和非阻塞是描述程序在等待调用结果（消息，返回值）时的状态。下面我们以某线程调用为例来解释两个概念。
阻塞：线程调用方法，其结果返回之前，当前线程会被挂起。调用线程只有在得到结果之后才会再次执行。
非阻塞：线程调用方法，在没有获得结果，会直接返回，不会一直等待结果，线程不会被挂起。

同步与异步

同步和异步是描述程序之间的信息通信机制的方式。
同步：当线程发出一个调用，在调用没有获得数据之前，此调用不会返回。
异步：当线程发出一个调用，此调用会立即返回，当被调用者准备好结果时，通过状态、通知来通知线程，或通过回调函数处理这个调用。

内部组件详解

NIO 内部的核心组件 Channel，Buffer 和 Selector。下面介绍这三个部分内容。

Channel

通道是原始 IO 流的抽象，其与流相似，但有如下的区别：

  • 通道可读可写，但流的读写通常是单向的。
  • 通道可以异步地读写。
  • 通道读写都必须与 Buffer 交互。

下面介绍 Channel 相关类的层级关系，使用缩进来表示这些类的层级关系。

  • AbstractInterruptibleChannel
    • FileChannel
    • SelectableChannel
      • AbstractSelectableChannel
        • DatagramChannel
        • SocketChannel
        • ServerSocketChannel

AbstractInterruptibleChannel：可中断通道的基本实现类。此抽象类封装了实现通道异步关闭和中断所需的低级别机制。

FileChannel：用于读取、写入、映射和操作文件的通道。这是一个抽象类，其内部定义了很多文件的操作方法。官方给出了如下的文件操作的特性描述：

  1. 对文件中绝对位置的字节进行读取或写入，对通道当前位置无影响。
  2. 将文件中的某个区域直接映射到内存中；对于较大的文件，这样的读写方法更为高效。
  3. 强制对底层存储设备进行文件的更新，确保在系统崩溃时不丢失数据。
  4. 以一种可被很多操作系统优化为直接向文件系统缓存发送或从中读取的高速传输方法，将字节从文件传输到某个其他通道中，反之亦然。
  5. 可以锁定某个文件区域，以阻止其他程序对其进行访问。

FileChannel 的实例化，可以通过其类中的 open 静态方法，同样也可以从现有的 FileInputStream、FileOutputStream 或 RandomAccessFile 对象获得文件通道，通过调用这些对象的 getChannel 方法，这会返回一个连接到相同底层文件的文件通道。FileChannel 不可以设置阻塞模式。

SelectableChannel：可利用 Selector 实现多路复用的通道。与选择器一起使用，此类的实例必须首先通过 register 方法进行注册。

AbstractSelectableChannel：可选择通道的基本实现类。此类定义了处理通道注册、注销和关闭机制的各种方法。它会维持此通道的当前阻塞模式及其当前的选择键集。

DatagramChannel：此类是针对面向数据报（UDP）套接字的可选择通道。通过 open 方法创建数据报通道。新创建的数据报通道已打开，但尚未连接。可以使用 send 和 receive 方法，无需连接数据报通道。如果为了避免作为每次发送和接收操作的一部分而执行的安全检查开销，可以通过调用 connect 方法来建立数据报通道连接。此时可以使用 read 和 write 方法进行数据的交互。

SocketChannel：针对面向流的连接套接字 (Client TCP)的可选择通道。可以用 open 方法创建套接字通道，此时套接字通道已打开，但尚未连接。若在未连接的通道中进行操作，会抛出 NotYetConnectedException。可通过调用套接字通道的 connect 方法连接该通道；一旦连接后，关闭套接字通道之前它会一直保持已连接状态。

方法名 功能 特征
open(SocketAddress remote) 打开套接字通道并将其连接到远程地址 调用 open() 方法、在得到的套接字通道上调用 connect 方法、向其传递 remote，然后返回该通道

write(ByteBuffer src) 将字节序列从给定的缓冲区写入此通道 尝试最多向该通道中写入 n 个字节，其中 n 是调用此方法时缓冲区中剩余的字节数，即 src.remaining()
socket() 获取与此通道关联的套接字 返回的对象不会声明任何在 Socket 类中未声明的公共方法
shutdownInput() 在不关闭通道的情况下，关闭读连接 关闭后，再读数据，则返回 -1
shutdownOutput() 在不关闭通道的情况下，关闭写连接 关闭后，再写数据，则抛出 ClosedChannelException

ServerSocketChannel：针对面向流的侦听套接字 (Server TCP) 的可选择通道。可以调用 open 方法创建服务器套接字通道。新创建的服务器套接字通道已打开，但尚未绑定。此时若直接调用 accept 方法会导致抛出 NotYetBoundException，需先调用服务器套接字的 bind 方法来绑定服务器套接字通道。

Buffer

Java NIO 中的 Buffer 用于和 NIO 通道进行双向交互的。下面我们介绍 NIO 包下所有的 buffer 相关的类。从类命名即可看出，这些 “*Buffer” 类内部是通过 byte，char，short，int，long，float 或 double 类型来操作缓冲区中的字节。

  • Buffer
    • ByteBuffer
      • MappedByteBuffer
    • CharBuffer
    • ShortBuffer
    • IntBuffer
    • LongBuffer
    • FloatBuffer
    • DoubleBuffer

Buffer: 它是所有 Buffer 相关类的基类，其提供了一系列的基本约定和操作方法。缓冲区是特定基本类型元素的线性有限序列。除内容外，缓冲区的基本属性还包括容量、限制、位置和标记。下面简要分析这四个属性。

  • capacity：是缓冲区所包含的元素的数量。缓冲区的容量不能为负。
  • limit：是第一个不应该读取或写入的元素的位置索引。缓冲区的限制不能为负，并且不能大于其容量。
  • position：是下一个要读取或写入的元素的位置索引。缓冲区的位置不能为负，并且不能大于其限制。
  • mark：在此缓冲区中标记当前的位置。对 reset 方法的后续调用会在最后标记的位置重新定位此流，以便后续读取重新读取相同的字节。

这四个属性会一直维持这样的关系：-1 <= mark <= position <= limit <= capacity。

下面列出其内部一些常用的方法：

方法名 功能 特征
flip() 反转此缓冲区，读写模式切换 限制设置为当前位置，然后将位置设置为 0。如果已定义了标记，则丢弃该标记
rewind() 重置此缓冲区 将位置置为 0 并丢弃标记
isReadOnly() 告知此缓冲区是否为只读缓冲区 虚方法，子类可选实现
array() 返回此缓冲区的底层实现数组 虚方法，子类可选实现
reset() 将此缓冲区的位置重置为之前标记的位置 如果尚未设置标记，会抛出 InvalidMarkException

ByteBuffer：继承自 Buffer，实现了 Comparable 泛型接口。它是针对字节这个类型的缓冲区实现，根据字节的特性，内部实现便利的字节操作方法。
其方法可归纳为以下几个部分：

  1. 获取和写入数据：单个字节，批量数据等。
  2. 读写其他基本类型的数据的方法，如 getChar，putChar。
  3. 创建视图缓冲区的方法，可将字节缓冲区视为包含其他基本类型值的缓冲区。
  4. 对字节缓冲区进行 compacting（压缩）、duplicating（共享整个序列） 和 slicing（共享子序列） 的方法。

关于创建视图缓冲区的优点有以下几点：

  • 视图缓冲区不是根据字节进行索引，而是根据其特定于类型的值的大小进行索引；
  • 视图缓冲区对特定类型提供了相对批量 get 和 put 方法；
  • 视图缓冲区可能更高效，仅当字符缓冲区为直接缓冲区，其才是直接缓冲区。

缓冲区可以分为直接缓冲区和非直接缓冲区。若是直接字节缓冲区，则 Java 虚拟机会可能的直接在此缓冲区上执行本机 I/O 操作。在具体操作中，虚拟机都会尽量避免将缓冲区的内容复制到中间缓冲区中。 直接缓冲区可以使用 allocateDirect 工厂方法来创建，其创建和取消分配的所需成本通常高于非直接缓冲区。所以仅当直接缓冲区能带来明显的好处时候才使用它。

Selector

Selector 提供了查询通道的 IO 操作是否准备好的能力。Selector 允许单线程处理多个 Channel。相对于一个线程一个通道来说，不仅仅减少了线程资源的使用，还节约了线程之间上下文切换的开销。

SelectionKey：表示 SelectableChannel 在 Selector 中的注册的标记。每次向选择器注册通道时就会创建一个选择键。调用某键的 cancel 方法、关闭其通道。取消键，会把该键添加到选择器的已取消键集，在下一次进行选择操作时移除它。
SelectionKey 主要含有下面四个部分：

  • Channel
  • Selector
  • interest 集合 :可以在通道注册选择器时对感兴趣的事务进行设置
  • ready 集合 ：是通道已经准备就绪的集合

四个 SelectionKey 常量代表四个不同的事件:

  1. SelectionKey.OP_CONNECT
  2. SelectionKey.OP_ACCEPT
  3. SelectionKey.OP_READ
  4. SelectionKey.

与传统 IO 的比较

阻塞 IO 与非阻塞 IO

按照概念介绍部分中给出的阻塞与非阻塞的概念，两者最大的区别：传统 IO 是阻塞的，NIO 是非阻塞的。两者都是同步的。

  • 传统 IO : 同步阻塞 IO
  • NIO : 同步非阻塞 IO

传统 IO 是同步阻塞的毋庸置疑，下面分析 NIO 同步和非阻塞的原因。
在 NIO 中，一个 read 操作通常包括两个不同阶段：

  1. 等待通道数据准备好；
  2. 从通道向缓冲复制数据。

根据《UNIX网络编程》中的 IO multiplexing model 图模拟了 Java NIO 中的图示：

从图中可知，程序会先调用 select 方法，NIO 提供了三种 Select 方法，如上表所示。当调用阻塞的 select 方法时，会程序会一直阻塞，直至通道数据准备完毕，当调用非阻塞的 select 方法时，程序会直接返回 0，然后再不断的调用此方法，直至通道数据准备完毕。当通道数据准备完毕后，会进行通道数据复制到缓冲的操作，此时程序是阻塞的。虽然程序是阻塞的，但这是由于调用自身的方法导致的，并不是因为 IO 操作导致的，所以称 NIO 为非阻塞的。
关于同步和异步，《UNIX网络编程》中有这样的一段话：

A synchronous I/O operation causes the requesting process to be blocked until that I/O operation completes.
An asynchronous I/O operation does not cause the requesting process to be blocked.

图中，明显看出程序在 NIO 中程序是肯定会阻塞的，所以称 NIO 为同步的。

面向流和面向缓冲

传统 IO 每次只能从流中一个一个字节读，直至读完所有字节，没有被缓存在任何地方。若需要前后移动流中的数据，需要将其先读取到一个缓冲区中。
NIO 数据全部被读取到一个稍后处理它的缓冲区中，可以移动，检查是否含有相关数据等，灵活性很高。

选择器

NIO 中可以将多个通道注册使用一个选择器，这样对于每个通道对应一个线程的方式来说，非常方便。
